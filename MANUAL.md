## 1. บทนำ

โปรแกรมนี้เป็นเครื่องมือตรวจสอบความถูกต้องของข้อมูล หลังจากที่มีการโยกย้าย (migrate) ฐานข้อมูลจาก MySQL 4.0 (ฐานข้อมูลต้นทาง) ไปยัง MySQL 8 (ฐานข้อมูลปลายทาง) มีไว้เพื่อยืนยันว่าข้อมูลที่ย้ายไปแล้วครบถ้วนและตรงกับต้นฉบับ ก่อนที่จะปิดระบบเก่าหรือให้ผู้ใช้งานจริงเริ่มใช้ระบบใหม่

โปรแกรมจะตรวจสอบให้ทั้งหมดในคราวเดียว ครอบคลุม:

- โครงสร้างตาราง (schema) — ชนิดข้อมูลและขนาดของแต่ละคอลัมน์
- จำนวนแถว (row count) ของแต่ละตาราง
- ดัชนี (index) และคีย์หลัก (primary key)
- คีย์หลักที่ซ้ำกัน (duplicate primary key)
- ค่าข้อมูลจริงในแต่ละแถว เทียบทีละฟิลด์ (field-by-field)

ผลการตรวจสอบจะแสดงผ่านหน้าเว็บแดชบอร์ดแบบเรียลไทม์ พร้อมดาวน์โหลดรายงานรายการที่ไม่ตรงกัน (mismatch) เป็นไฟล์ Excel หรือ CSV เพื่อนำไปตรวจสอบต่อได้

หมายเหตุ: คู่มือนี้เขียนขึ้นสำหรับผู้ที่จะใช้งานโปรแกรมนี้จริง ไม่จำเป็นต้องมีความรู้ด้านการเขียนโปรแกรม แต่ควรมีความคุ้นเคยกับคำศัพท์พื้นฐานของฐานข้อมูล เช่น ตาราง (table), แถว (row) และคอลัมน์ (column)

## 2. สิ่งที่ต้องเตรียมก่อนใช้งาน

- เครื่องคอมพิวเตอร์ที่ลง Python เวอร์ชัน 3.10 ขึ้นไป
- สิทธิ์เข้าถึงฐานข้อมูล ทั้งฝั่งต้นทาง (MySQL 4.0) และปลายทาง (MySQL 8) — ใช้บัญชีที่มีสิทธิ์ "อ่านอย่างเดียว" (read-only) ก็เพียงพอ เพราะโปรแกรมนี้ไม่มีการเขียนหรือแก้ไขข้อมูลใดๆ ในฐานข้อมูล
- การเชื่อมต่อเครือข่าย หากใช้งานจากนอกเครือข่ายบริษัท ต้องเชื่อมต่อ VPN ก่อน จึงจะเข้าถึงฐานข้อมูลทั้งสองฝั่งได้
- ไลบรารีสำหรับเชื่อมต่อ MySQL (ใช้ตอนติดตั้งโปรแกรมเท่านั้น):
  - macOS: `brew install mysql-client pkg-config`
  - Ubuntu/Debian: `sudo apt install default-libmysqlclient-dev build-essential pkg-config`
  - Windows: ไม่ต้องติดตั้งอะไรเพิ่มเติม การติดตั้งไลบรารี Python ในขั้นตอนถัดไปจะจัดการให้อัตโนมัติ

## 3. การติดตั้งโปรแกรม

1. ดาวน์โหลดโค้ดโปรเจกต์ลงเครื่อง:
   ```
   git clone https://github.com/NuramX/mysql4to8_migration_test_script.git
   cd mysql4to8_migration_test_script
   ```
2. สร้าง virtual environment (พื้นที่แยกสำหรับติดตั้งไลบรารีของโปรเจกต์นี้โดยเฉพาะ):
   ```
   python3 -m venv .venv
   ```
3. เปิดใช้งาน virtual environment:
   ```
   source .venv/bin/activate   # สำหรับ Windows ใช้: .venv\Scripts\activate (รันใน cmd)
   ```
4. ติดตั้งไลบรารีที่จำเป็นทั้งหมด:
   ```
   pip install -r requirements.txt
   ```
   หากติดตั้งไม่ผ่าน: ดูหัวข้อ "9. การแก้ปัญหาที่พบบ่อย" ท้ายคู่มือ ส่วนใหญ่เกิดจากยังไม่ได้ลงไลบรารี MySQL client ตามหัวข้อที่ 2

## 4. การตั้งค่าการเชื่อมต่อฐานข้อมูล

ก่อนใช้งาน ต้องบอกโปรแกรมก่อนว่าฐานข้อมูลต้นทางและปลายทางอยู่ที่ไหน เข้าด้วยบัญชีอะไร เลือกได้ 2 วิธี ทำวิธีใดวิธีหนึ่งก็พอ:

### 4.1 ตั้งค่าผ่านหน้าเว็บ (แนะนำ ง่ายที่สุด)

วิธีนี้ไม่ต้องสร้างไฟล์ตั้งค่าเอง เหมาะกับผู้ใช้งานทั่วไป

1. เปิดโปรแกรมด้วยคำสั่ง: `python web_server.py`
2. เปิดเบราว์เซอร์ไปที่ `http://localhost:5001`
3. คลิกปุ่ม "⚙️ Connections" มุมขวาบนของหน้าจอ เพื่อเปิดแผงตั้งค่าการเชื่อมต่อ
4. กรอกข้อมูล host, port, user และ password ของฐานข้อมูลต้นทาง (Source) และปลายทาง (Target)
5. กดปุ่ม "Test & Save Target" ที่ตัว Target databases ก่อน จากนั้นเมื่อ connect ได้ ค่อยกด "Test & Save Connection Settings" ที่ด้านล่างอีกที

ระบบจะทดสอบเชื่อมต่อฐานข้อมูลทั้งสองฝั่งก่อนบันทึกเสมอ หากเชื่อมต่อไม่สำเร็จ ระบบจะแจ้งข้อผิดพลาดและจะไม่บันทึกค่าที่กรอกไว้ เมื่อบันทึกสำเร็จ ข้อมูลจะถูกเขียนลงไฟล์ `config.json` ให้อัตโนมัติ

### 4.2 ตั้งค่าผ่านไฟล์ config.json (จำเป็นหากจะใช้งานผ่าน Command Line)

1. คัดลอกไฟล์ตัวอย่างเป็นไฟล์ใช้งานจริง:
   ```
   cp config.example.json config.json
   ```
2. เปิดไฟล์ `config.json` แล้วแก้ไขค่าตามตารางด้านล่าง

| ค่า | ความหมาย |
|---|---|
| `source` | ข้อมูลเชื่อมต่อฐานข้อมูล MySQL 4.0 (ต้นทาง): host, port, user, password |
| `target` | ค่าที่ระบบ sync ให้อัตโนมัติจาก `targets[]` + `active_target` ทุกครั้งที่บันทึก — **ห้ามแก้เอง** |
| `targets` | ข้อมูลเชื่อมต่อฐานข้อมูลปลายทาง (รองรับได้หลายตัว) |
| `targets[].name` | ชื่อเรียก target แต่ละตัว (ตั้งเอง, ห้ามซ้ำ) ใช้ผูกกับ `active_target` เพื่อเลือกว่าจะ compare กับ target ตัวไหน |
| `targets[].host` | Host/IP ของ database |
| `targets[].port` | Port ของ database |
| `targets[].username` | Username ของ database |
| `targets[].password` | Password ของ database |
| `targets[].version` | Version ของ mysql database ว่าเป็น 4 หรือ 8 |
| `active_target` | ชื่อ target (ต้องตรงกับ `targets[].name` ตัวใดตัวหนึ่ง) ที่ระบบจะใช้เป็นปลายทางเปรียบเทียบข้อมูลอยู่ตอนนี้ |
| `default_db` | ชื่อฐานข้อมูลที่เปิดมาให้เป็นค่าเริ่มต้นในหน้าแดชบอร์ด |
| `tuning.data_window_years` | ตรวจเฉพาะข้อมูล N ปีปฏิทินล่าสุด (นับจากคอลัมน์วันที่ในคีย์หลัก) เช่นใส่ 2 ในปี 2569 = ตรวจข้อมูลตั้งแต่ 1 ม.ค. 2567 เป็นต้นไป ตารางที่ไม่มีคอลัมน์วันที่จะตรวจข้อมูลทั้งตารางเสมอ ใส่ 0 = ปิดการกรอง ตรวจข้อมูลทุกตารางแบบเต็ม |
| `tuning.workers` | จำนวน thread ที่ตรวจสอบพร้อมกัน (ค่าแนะนำ 8) |
| `tuning.src_timeout` | ระยะเวลารอผลคำสั่ง query ฝั่งต้นทางก่อนตัดการเชื่อมต่อ (วินาที) |
| `tuning.max_errors` | จำนวนรายการที่ไม่ตรงกัน (mismatch) สูงสุดที่จะเก็บลงรายงานต่อหนึ่งตาราง |
| `tuning.decimal_tol` | ค่าความคลาดเคลื่อนของตัวเลขทศนิยมที่ยอมรับได้ระหว่างสองฝั่ง (มีผลเมื่อไม่ได้ตั้ง `decimal_round`) |
| `tuning.decimal_round` | ปัดเศษทศนิยมทั้งสองฝั่งเป็น N ตำแหน่งก่อนเปรียบเทียบ หากตั้งค่านี้จะมีผลแทน `decimal_tol` ตั้งเป็น `null` เพื่อปิด (ตั้งจากหน้าเว็บ Settings ได้เช่นกัน) |
| `tuning.hash_chunks` | จำนวนส่วนย่อย (chunk) ที่ใช้ตอนคำนวณ checksum เพื่อเทียบข้อมูล |
| `ignore_fields` | ระบุคอลัมน์ที่ต้องการข้ามไม่ต้องเทียบ แยกตามฐานข้อมูลและตาราง (เช่น คอลัมน์ `updated_at` ที่เปลี่ยนค่าระหว่างการ migrate ตามปกติ) |

> **ข้อควรระวัง:** ห้าม commit ไฟล์ `config.json` ขึ้น git เด็ดขาด เพราะมีรหัสผ่านจริงของฐานข้อมูลอยู่ในไฟล์นี้

### 4.3 การตั้งค่า ts_field_config.json (คอลัมน์วันที่หลักของแต่ละตาราง)

ไฟล์นี้บอกให้ระบบรู้ว่า "แต่ละตารางใช้คอลัมน์ไหนเป็นคอลัมน์วันที่/เวลาหลัก" ค่าที่ตั้งไว้ในไฟล์นี้มีผลกับโปรแกรม 3 จุด:

- ตัวกรอง Data Range ในหน้า Overview (ดูหัวข้อ 5.3) — เวลาตั้งช่วงปี/เดือน/วัน หรือใช้ค่า `data_window_years` จาก config.json ระบบจะใช้คอลัมน์ที่ตั้งไว้ในไฟล์นี้เป็นตัวกรองว่าจะดึงแถวไหนของตารางนั้นมาตรวจ
- การ์ด "Timestamp Range & Row Count" ในหน้ารายละเอียดตาราง (ดูหัวข้อ 5.9) — แสดงค่าน้อยสุด/มากสุด (MIN/MAX) ของคอลัมน์นี้ เพื่อยืนยันขอบเขตของข้อมูลที่ถูกตรวจ
- รายงานรายการที่ขาด/เกิน (Missing / Extra) — เวลาพบแถวที่มีอยู่แค่ฝั่งใดฝั่งหนึ่ง ระบบจะแสดงค่าคอลัมน์นี้กำกับไว้ในรายงาน ช่วยให้หาตำแหน่งของข้อมูลที่ขาดหายได้เร็วขึ้น

รูปแบบไฟล์เป็น JSON ซ้อนกัน 2 ชั้น: ชื่อฐานข้อมูล → ชื่อตาราง → ชื่อคอลัมน์ ตัวอย่าง:

```json
{
  "prs": {
    "daily_price": "timestamp",
    "NH": "TimeStamp",
    "counter": "date"
  }
}
```

วิธีแก้ไข:

1. เปิดไฟล์ `ts_field_config.json` ด้วยโปรแกรมแก้ไขข้อความทั่วไป (เช่น Notepad, VS Code)
2. หาบล็อกของฐานข้อมูลที่ต้องการ (ชื่อฐานข้อมูลเป็น key ชั้นนอกสุด) ถ้ายังไม่มีให้เพิ่มบล็อกใหม่
3. เพิ่มหรือแก้ไขบรรทัดในรูปแบบ `"ชื่อตาราง": "ชื่อคอลัมน์"` — ชื่อคอลัมน์ต้องสะกดตรงกับชื่อจริงในฐานข้อมูลทุกตัวอักษร รวมถึงตัวพิมพ์เล็ก/ใหญ่ (เช่นบางตารางใช้ `TimeStamp` บางตารางใช้ `timestamp` ตัวเล็กล้วน ต้องเช็คของจริงในตารางนั้นก่อนใส่)
4. บันทึกไฟล์

**ไม่ต้อง restart:** ไม่ต้องปิดหรือรีสตาร์ท `web_server.py` หลังแก้ไฟล์นี้ ระบบจะอ่านไฟล์ใหม่ทุกครั้งที่กด Run Validation, เปิดหน้ารายละเอียดตาราง หรือรัน SQL Compare

ตารางที่ยังไม่ได้ตั้งค่าไว้ในไฟล์นี้:

- ระบบจะพยายามหาคอลัมน์ในคีย์หลัก (primary key) ที่เป็นชนิดวันที่/เวลาให้อัตโนมัติแทน
- ถ้าหาไม่เจอเลย ตารางนั้นจะถูกตรวจข้อมูลเต็มทั้งตารางเสมอ (ไม่ถูกกรองตาม Data Range) และการ์ด Timestamp Range จะแสดงแค่จำนวนแถว (Row Count) แทน โดยไม่มีช่วงวันที่ให้ดู

> **ข้อควรระวัง:** ถ้าใส่ชื่อคอลัมน์ผิด หรือคอลัมน์นั้นไม่มีอยู่จริงในตาราง ระบบจะฟ้อง error (เช่น Unknown column) แทนที่จะแสดงผลตามปกติ ให้ตรวจสอบชื่อคอลัมน์ให้ตรงกับฐานข้อมูลจริงก่อนบันทึกเสมอ นอกจากนี้ควรเลือกคอลัมน์ที่มี index อยู่แล้ว เพราะระบบใช้คำสั่งเรียงลำดับ (`ORDER BY … LIMIT 1`) เพื่อหาค่าน้อยสุด/มากสุด หากคอลัมน์ไม่มี index จะทำให้ตรวจช้าลงมากในตารางขนาดใหญ่

## 5. การใช้งานหน้าเว็บแดชบอร์ด (วิธีหลักที่แนะนำ)

### 5.1 เปิดโปรแกรม

รันคำสั่งต่อไปนี้จากโฟลเดอร์โปรเจกต์: `python web_server.py` แล้วเปิดเบราว์เซอร์ไปที่ `http://localhost:5001`

### 5.2 เลือกฐานข้อมูลที่ต้องการตรวจสอบ

มุมบนซ้ายของหน้าจอมีเมนูดรอปดาวน์สำหรับเลือกฐานข้อมูล (database) เมื่อเลือกแล้ว ระบบจะแสดงป้าย (badge) บอกเวอร์ชันและ host ของฝั่งต้นทาง (Source) และปลายทาง (Target) กำกับไว้ด้านบนให้ตรวจสอบว่าถูกต้อง ก่อนเริ่มการตรวจสอบ

### 5.3 ตั้งค่าตัวเลือกก่อนเริ่มตรวจสอบ (ไม่บังคับ)

- **Skip Full Sync (fast mode)** — ค่าเริ่มต้นติ๊กไว้ ใช้เพื่อข้ามการเทียบข้อมูลแบบละเอียดทีละแถว ตรวจเฉพาะโครงสร้างและจำนวนแถวเพื่อความรวดเร็ว หากต้องการตรวจข้อมูลจริงแบบเต็มทุกแถว ให้เอาเครื่องหมายถูกออก (ใช้เวลานานกว่ามาก ขึ้นกับขนาดข้อมูล)
- **Auto Random Sample** — ติ๊กเพื่อให้ระบบสุ่มเทียบข้อมูลรายแถว (field-by-field) ให้ทุกตารางโดยอัตโนมัติทันทีหลังตรวจสอบเสร็จ โดยไม่ต้องกดเข้าไปดูทีละตารางเอง
- **Data Range** — กำหนดช่วงปี/เดือน/วันของข้อมูลที่จะตรวจ (อ้างอิงจากคอลัมน์วันที่ในคีย์หลักของแต่ละตาราง) หากไม่ระบุ ระบบจะใช้ค่าเริ่มต้นจาก `data_window_years` ใน config.json

### 5.4 เริ่มตรวจสอบ

กดปุ่ม "Run Validation" มุมขวาบน ระบบจะเริ่มตรวจสอบทุกตารางในฐานข้อมูลที่เลือกไว้ตามลำดับ และแสดงผลแบบเรียลไทม์ทีละตารางโดยไม่ต้องรีเฟรชหน้าจอ (ไม่แนะนำใช้ตัวนี้ในการตรวจข้อมูล แนะนำให้ใช้ feature ในข้อ 5.8 ในการตรวจ จะทำให้การตรวจมี performance ที่ดีกว่าสำหรับ tester)

### 5.5 อ่านผลสรุปภาพรวม

ระหว่างและหลังการตรวจสอบ จะเห็นแถบสรุปจำนวนตารางแยกตามสถานะ:

- **Tables Pass** — ตารางที่ผ่านการตรวจสอบทุกรายการ
- **Tables Fail** — ตารางที่พบปัญหาซึ่งต้องตรวจสอบและแก้ไข
- **Warnings** — ตารางที่มีข้อสังเกตควรตรวจสอบเพิ่มเติม แต่ยังไม่ถือว่าล้มเหลว
- **Skipped** — ตารางที่ยังไม่ได้ตรวจสอบ (เช่น กดหยุดกลางคัน)

หากมีตารางที่มีอยู่ในฝั่งต้นทางแต่ไม่พบในฝั่งปลายทาง ระบบจะแจ้งเตือนด้วยข้อความ "⚠ Tables missing in target" พร้อมจำนวนตารางที่ขาดหายไป ซึ่งต้องตรวจสอบเป็นลำดับแรก เพราะหมายถึงข้อมูลทั้งตารางยังไม่ถูกย้าย

### 5.6 อ่านผลลัพธ์รายตาราง

ตารางผลลัพธ์แสดงหนึ่งแถวต่อหนึ่งตารางข้อมูล พร้อมคอลัมน์การตรวจสอบแต่ละประเภท แต่ละช่องจะแสดงสัญลักษณ์ต่อไปนี้:

| สัญลักษณ์ | สถานะ | ความหมาย |
|---|---|---|
| ✓ (สีเขียว) | PASS | ผ่านการตรวจสอบ |
| ✗ (สีแดง) | FAIL | พบปัญหา ต้องตรวจสอบและแก้ไข |
| ! (สีเหลือง) | WARN | มีข้อสังเกต ควรตรวจสอบเพิ่มเติม |
| – (สีเทา) | SKIP | ยังไม่ได้ตรวจสอบรายการนี้ |

คอลัมน์การตรวจสอบแต่ละประเภทมีความหมายดังนี้:

| คอลัมน์ | ตรวจสอบอะไร |
|---|---|
| Row Count Check | เทียบจำนวนแถวระหว่างต้นทางและปลายทาง ว่ามีรายการตกหล่นหรือไม่ |
| Schema Check | เช็คความเข้ากันได้ของฟิลด์ ขนาด และชนิดข้อมูล ว่าปลายทางรองรับถูกต้องหรือไม่ |
| Index Check | ตรวจสอบความครบถ้วนของ Index และ Primary Key ทั้งหมดของตาราง |
| Duplicate PK Check | เช็คความไม่ซ้ำกัน (uniqueness) ของฟิลด์ Primary Key เพื่อความสมบูรณ์ของข้อมูล |
| Full Sync Check | เปรียบเทียบข้อมูลแบบละเอียดทีละแถว ทุกฟิลด์ในทุกแถว (100%) ว่าตรงกันจริงหรือไม่ |
| Error Summary | สรุปข้อความปัญหาที่พบโดยย่อ สำหรับตารางที่ไม่ผ่าน |

หมายเหตุ: หาก Skip Full Sync ถูกติ๊กไว้ (ค่าเริ่มต้น) คอลัมน์ Full Sync Check จะแสดง SKIP จนกว่าจะสั่งตรวจข้อมูลแบบละเอียดเพิ่มเติม (ดูหัวข้อ 5.8 หรือ 5.9)

### 5.7 กรองผลลัพธ์

ใช้ปุ่ม ALL / FAIL / WARN / PASS เหนือตารางผลลัพธ์ เพื่อกรองแสดงเฉพาะตารางตามสถานะที่ต้องการ เช่น กด FAIL เพื่อดูเฉพาะตารางที่มีปัญหาต้องแก้ไขก่อน

### 5.8 ตรวจสอบซ้ำเฉพาะบางตาราง (Quick Scan / Full Sync)

หากต้องการตรวจสอบซ้ำเฉพาะบางตาราง โดยไม่ต้องรัน Run Validation ใหม่ทั้งฐานข้อมูล:

1. ติ๊กเลือกตารางที่ต้องการ จาก checkbox ทางซ้ายสุดของแต่ละแถว (ใช้ "Select All Visible" เพื่อเลือกตารางที่กำลังแสดงตามตัวกรองทั้งหมด หรือ "Deselect All" เพื่อยกเลิก)
2. เลือกการตรวจสอบที่ต้องการ:
   - **Quick Scan Selected** — ตรวจใหม่เฉพาะโครงสร้างตารางและจำนวนแถวของตารางที่เลือก (ทำงานเร็ว)
   - **Run Full Sync on Selected** — ตรวจเทียบข้อมูลแบบละเอียดทีละแถวเฉพาะตารางที่เลือก (ละเอียดกว่าแต่ใช้เวลานานกว่า)

### 5.9 ดูรายละเอียดเชิงลึกของแต่ละตาราง

คลิกที่แถวของตารางใดก็ได้ในตารางผลลัพธ์ เพื่อเปิดหน้าต่างรายละเอียด ซึ่งจะแสดงข้อมูลดังนี้:

- **โครงสร้างตาราง (Schema)** — เทียบคอลัมน์และชนิดข้อมูลระหว่างต้นทางและปลายทาง
- **ช่วงเวลาของข้อมูล (Timestamp Range) และจำนวนแถว** — แสดงวันที่ต่ำสุด/สูงสุดของข้อมูลในตาราง เพื่อยืนยันขอบเขตของข้อมูลที่ถูกตรวจ
- **ผลการเทียบข้อมูลแบบละเอียด (Full Field Sync Details)** — สรุปผลการเทียบข้อมูลทีละแถวทีละฟิลด์
- **ตัวสำรวจรายการที่ไม่ตรงกัน (Mismatches Explorer)** — แสดงรายการที่ไม่ตรงกันทั้งหมด กรองแยกได้เป็น MISMATCH (ค่าต่างกัน) / MISSING (มีในต้นทางแต่ไม่มีในปลายทาง) / EXTRA (มีในปลายทางแต่ไม่มีในต้นทาง) คลิกแต่ละแถวเพื่อดูค่าที่ต่างกันทีละฟิลด์
- **การสุ่มเทียบข้อมูลรายแถว (Random Field-by-Field Comparison)** — สั่งสุ่มตัวอย่างข้อมูลมาเทียบเพิ่มเติมได้จากหน้านี้โดยตรง โดยไม่ต้องรัน Full Sync ใหม่ทั้งตาราง

### 5.10 ยกเว้นคอลัมน์ที่ไม่ต้องการเทียบ (Ignore Field)

บางคอลัมน์ไม่จำเป็นต้องเทียบค่าให้ตรงกัน เช่น คอลัมน์ที่บันทึกเวลาแก้ไขล่าสุด (`updated_timestamp`) ซึ่งเปลี่ยนค่าเป็นปกติระหว่างขั้นตอน migrate หรือคอลัมน์ที่รู้อยู่แล้วว่าปลายทางจงใจเก็บค่าต่างจากต้นทาง กรณีแบบนี้สามารถสั่งให้ระบบข้ามคอลัมน์นั้นได้โดยไม่ต้องแก้ไฟล์ config.json เอง:

1. เปิดหน้ารายละเอียดของตาราง (คลิกที่แถวตารางตามหัวข้อ 5.9 ทำหลังจาก quick scan หรือ full sync แล้ว 1 รอบ เพื่อให้มันรู้ว่า database มี field อะไรอยู่บ้าง)
2. ในตาราง "โครงสร้างตาราง (Schema)" ส่วนบนสุด ติ๊กเครื่องหมายถูกที่ช่อง "Ignore" ของคอลัมน์ที่ต้องการข้าม
3. ระบบจะบันทึกลง `config.json` (ในส่วน `ignore_fields`) ให้อัตโนมัติทันที และแสดงแถวของคอลัมน์นั้นเป็นสีจาง (ignored) ในตารางเดียวกัน

การตั้งค่านี้จะมีผลกับการตรวจสอบครั้งถัดไป (Run Validation, Quick Scan หรือ Full Sync รอบใหม่) — ผลตรวจสอบรอบที่รันไปแล้วก่อนหน้าจะไม่เปลี่ยนแปลงย้อนหลัง คอลัมน์ที่เป็นคีย์หลัก (Primary Key) จะไม่สามารถติ๊กข้ามได้ เพราะระบบต้องใช้คีย์หลักในการจับคู่แถวเสมอ หากต้องการยกเลิกการข้าม ให้เอาเครื่องหมายถูกออกจากช่อง Ignore ในหน้าเดิมได้ทันที

### 5.11 ดาวน์โหลดรายงาน

ในหน้ารายละเอียดของตารางที่พบปัญหา สามารถดาวน์โหลดรายงานรายการที่ไม่ตรงกันได้ 2 รูปแบบ:

- ไฟล์ Excel (.xlsx) — จัดรูปแบบพร้อมเปิดตรวจสอบและส่งต่อทีมงาน
- ไฟล์ CSV ดิบ — สำหรับนำไปประมวลผลต่อด้วยเครื่องมืออื่น

> **ข้อควรระวัง:** ไฟล์รายงานทั้งหมดถูกเก็บไว้ที่โฟลเดอร์ `static/reports/` บนเครื่องที่รันโปรแกรม และจะถูกลบ/เขียนทับใหม่ทุกครั้งที่เริ่ม Run Validation รอบใหม่ หากต้องการเก็บรายงานไว้อ้างอิง ให้ดาวน์โหลดออกมาเก็บไว้นอกโปรแกรมก่อน

### 5.12 สลับฐานข้อมูล

สามารถเปลี่ยนฐานข้อมูลที่ตรวจสอบได้จากดรอปดาวน์บนสุดตลอดเวลา โดยไม่ต้องกังวลว่าผลตรวจสอบเดิมจะหายไป เพราะระบบเก็บผลลัพธ์แยกต่างหากไว้ตามฐานข้อมูลแต่ละตัว

## 6. เครื่องมือเสริมสำหรับตรวจสอบเชิงลึก

นอกจากการตรวจสอบอัตโนมัติแล้ว หน้าเว็บยังมีเครื่องมือสำหรับตรวจสอบข้อมูลเฉพาะจุดที่การตรวจสอบอัตโนมัติไม่ครอบคลุม

### 6.1 SQL Runner — รันคำสั่งค้นหาข้อมูลเอง

ใช้สำหรับรันคำสั่ง SQL ประเภท SELECT เท่านั้น (อ่านข้อมูลอย่างเดียว ไม่สามารถแก้ไขหรือลบข้อมูลได้) กับฐานข้อมูลต้นทางหรือปลายทาง เหมาะสำหรับตรวจสอบข้อมูลที่สงสัยแบบเจาะจง

1. กดปุ่ม "SQL Runner" บนแถบด้านบนของหน้าจอ
2. เลือกฐานข้อมูลปลายทาง (target) ที่ต้องการรันคำสั่ง
3. เลือกจำนวนแถวสูงสุดที่ต้องการแสดงผล (limit)
4. พิมพ์คำสั่ง SELECT ในช่องข้อความ
5. กดปุ่ม "Run Query" เพื่อดูผลลัพธ์

### 6.2 SQL Compare — เทียบข้อมูลแบบกำหนดเงื่อนไขเอง

ใช้เทียบข้อมูลแบบละเอียดทีละฟิลด์ (field-by-field) ระหว่างต้นทางและปลายทาง โดยกำหนดเงื่อนไข SQL เอง เหมาะสำหรับกรณีที่ต้องการเทียบข้อมูลเฉพาะช่วงหรือเงื่อนไขพิเศษที่การตรวจสอบอัตโนมัติทั้งตารางไม่ครอบคลุม

1. กดปุ่ม "⚖ SQL Compare" บนแถบด้านบนของหน้าจอ
2. ค้นหาและเลือกตารางที่ต้องการเทียบ จากช่องค้นหา
3. เลือกคอลัมน์ที่ต้องการเทียบ (ต้องหยิบ pk มาให้ครบเสมอ ไม่งั้นจะไม่ให้ query)
4. เขียนเงื่อนไข SQL เพิ่มเติม (เช่น WHERE) ในช่องข้อความ
5. กดปุ่ม "Run Compare" เพื่อเริ่มเทียบข้อมูล (กด "Stop" ได้หากต้องการหยุดกลางคัน)
6. ดูผลลัพธ์แยกเป็น MATCH / MISMATCH / MISSING / EXTRA และดาวน์โหลดรายงาน Excel ได้เช่นเดียวกับหน้ารายละเอียดตาราง

## 7. การใช้งานผ่าน Command Line

เหมาะสำหรับทีมเทคนิคที่ต้องการรันตรวจสอบแบบอัตโนมัติ (เช่น ตั้งเวลารันอัตโนมัติ) โดยไม่ต้องเปิดหน้าเว็บ ต้องมีไฟล์ `config.json` อยู่แล้ว (ดูหัวข้อ 4.2)

```
python validate_all_tables.py --db <ชื่อฐานข้อมูล>
```

| flag | ความหมาย |
|---|---|
| `--db <ชื่อ>` | ระบุฐานข้อมูลที่จะตรวจสอบ (ถ้าไม่ใส่ จะใช้ `default_db` จาก config.json) |
| `--json` | พิมพ์ผลลัพธ์เป็น JSON ทีละบรรทัด (NDJSON) แทนข้อความอ่านง่ายปกติ เหมาะสำหรับต่อกับระบบอื่น |
| `--skip-sync` | ข้ามขั้นตอนเทียบข้อมูลแบบละเอียด ตรวจสอบเฉพาะโครงสร้างตารางและจำนวนแถวเท่านั้น |
| `--sync-limit <N>` | จำกัดจำนวนแถวสูงสุดที่จะเทียบข้อมูลแบบละเอียดต่อหนึ่งตาราง |
| `--tables <ชื่อ1,ชื่อ2,...>` | ระบุเฉพาะบางตารางที่ต้องการตรวจสอบ (คั่นด้วยจุลภาค) แทนที่จะตรวจสอบทั้งฐานข้อมูล |

## 9. การแก้ปัญหาที่พบบ่อย

**ติดตั้ง `pip install mysqlclient` ไม่สำเร็จ**
แปลว่ายังไม่ได้ลง MySQL client library ให้ดูหัวข้อ "2. สิ่งที่ต้องเตรียมก่อนใช้งาน" หากใช้ macOS แล้วยังไม่ผ่าน ให้ลองตั้งค่า path เพิ่มเติมด้วยคำสั่ง:
```
export PKG_CONFIG_PATH="$(brew --prefix mysql-client)/lib/pkgconfig"
```

**ขึ้นข้อความ `ERROR: cannot load config.json`**
แปลว่ายังไม่ได้สร้างไฟล์ `config.json` หรือรูปแบบ JSON ในไฟล์ผิดพลาด ให้ตรวจสอบตามหัวข้อ 4.2 อีกครั้ง หรือใช้วิธีตั้งค่าผ่านหน้าเว็บแทน (หัวข้อ 4.1)

**เชื่อมต่อฐานข้อมูลต้นทางไม่ได้ หรือค้าง (timeout)**
ตรวจสอบว่าเชื่อมต่อ VPN แล้วหรือยัง และไฟร์วอลล์อนุญาตให้เข้าถึง host ฝั่งต้นทางหรือไม่ หากยังไม่ได้ผล ลองเพิ่มค่า `src_timeout` ใน config.json ให้มากขึ้น

**พอร์ต 5001 ถูกใช้งานอยู่แล้ว (port ชนกัน)**
แก้ไขเลขพอร์ตได้ที่บรรทัดสุดท้ายของไฟล์ `web_server.py`

## 10. ข้อควรระวังด้านความปลอดภัยของข้อมูล

- ห้าม commit ไฟล์ `config.json` ขึ้น git เพราะมีรหัสผ่านจริงของฐานข้อมูลอยู่ในไฟล์
- ใช้บัญชีฐานข้อมูลสิทธิ์ "อ่านอย่างเดียว" (read-only) เท่านั้น โปรแกรมนี้ไม่มีความจำเป็นต้องเขียนหรือแก้ไขข้อมูลในฐานข้อมูลใดๆ
- รายงานใน `static/reports/` ถูกเขียนทับทุกครั้งที่ Run Validation ใหม่ หากต้องการเก็บรายงานไว้อ้างอิงในอนาคต ให้ดาวน์โหลดออกมาเก็บนอกโปรแกรมก่อนเริ่มตรวจสอบรอบใหม่

## 11. ติดต่อขอความช่วยเหลือเพิ่มเติม

หากพบปัญหาที่ไม่ได้อยู่ในคู่มือนี้ ให้ติดต่อทีมพัฒนา/ผู้ดูแลโปรเจกต์ การแนบข้อความ error และภาพหน้าจอประกอบจะช่วยให้ตรวจสอบและแก้ไขปัญหาได้รวดเร็วยิ่งขึ้น
